# Firestore Database Schema

## Overview

The database uses **Cloud Firestore** (NoSQL, document-oriented) with a flat collection structure. Articles published by users are stored in a top-level `articles` collection. Media files (thumbnails) are stored separately in **Firebase Cloud Storage**.

---

## Collection: `articles`

Each document in this collection represents a single article published by a user.

### Document ID
Auto-generated by Firestore (`collection.add()`). Using auto-generated IDs avoids write contention on sequential IDs and scales horizontally without coordination.

### Fields

| Field         | Type        | Required | Description                                                                 |
|---------------|-------------|----------|-----------------------------------------------------------------------------|
| `title`       | `string`    | Yes      | Article headline. Max 200 characters, enforced by security rules.           |
| `authorId`    | `string`    | Yes      | Firebase Auth UID of the user who published the article.                    |
| `publishedAt` | `timestamp` | Yes      | Server-side creation timestamp (`Timestamp.now()`).                         |
| `author`      | `string`    | No       | Display name of the author, if available.                                   |
| `description` | `string`    | No       | Short summary or subtitle of the article.                                   |
| `content`     | `string`    | No       | Full body text of the article.                                              |
| `url`         | `string`    | No       | External reference URL, if the article is based on a source.                |
| `urlToImage`  | `string`    | No       | Public download URL from Firebase Cloud Storage for the article thumbnail.  |

### Example Document

```json
{
  "title": "Breaking News: Flutter is Awesome",
  "authorId": "abc123uid",
  "author": "Eduardo G",
  "publishedAt": "2026-03-11T14:30:00Z",
  "description": "A deep dive into why Flutter is taking over mobile development.",
  "content": "Flutter, developed by Google, has become one of the most popular...",
  "urlToImage": "https://firebasestorage.googleapis.com/v0/b/symmetry-news-eg.appspot.com/o/media%2Farticles%2F1741701000000_photo.jpg?alt=media"
}
```

## Cloud Storage Structure

```
media/
└── articles/
    ├── 1741701000000_photo.jpg
    ├── 1741702000000_image.png
    └── ...
```

Files are named with a millisecond timestamp prefix to guarantee uniqueness and allow chronological sorting at the storage level.

---

## Indexes

### Composite Index

| Collection | Fields                                    | Purpose                                           |
|------------|-------------------------------------------|---------------------------------------------------|
| `articles` | `authorId ASC`, `publishedAt DESC`        | Query articles by a specific author, newest first |

Single-field indexes on `publishedAt` are managed automatically by Firestore and do not need to be defined explicitly.

---

## Design Decisions

### Why a flat `articles` collection instead of subcollections?

Articles are not inherently owned by a single user in the app's reading experience — all users can read all articles. A flat collection allows querying the global feed (`orderBy publishedAt`) without requiring a `collectionGroup` query, which simplifies both the client code and the security rules.

### Why optional fields are not enforced as required in the schema?

Only `title`, `authorId`, and `publishedAt` are required because they are the minimum viable data to display an article in the feed and enforce ownership. All other fields (`content`, `urlToImage`, etc.) are optional to support quick "headline-only" posts and reduce friction for the author.

### Why `publishedAt` uses a Firestore `Timestamp` instead of a string?

Using a native `Timestamp` type allows Firestore to sort and index the field efficiently. String-based dates would require the client to parse and sort, and would break ordering if date formats are inconsistent. The client converts `Timestamp` to `ISO 8601` string only at the data layer boundary, keeping the domain layer format-agnostic.

### Why anonymous authentication?

The backend uses Firebase Anonymous Auth as an intentional trade-off to satisfy the `authorId` requirement in security rules without forcing users to create an account. This allows frictionless publishing while still enforcing ownership rules (only the author can edit/delete their article). The authentication model is designed so that the anonymous session can later be upgraded to a permanent account (for example, via email or Google sign-in) without breaking existing data.
